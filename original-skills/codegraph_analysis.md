# 🕸️ 代码图谱静态分析与依赖图构建技能指南 (CodeGraph Analysis Skill)

本技能指南旨在指导 AI 助手如何系统性地对一个未知或复杂的软件项目进行**静态分析（Static Analysis）**、**依赖拓扑解析（Dependency Mapping）**以及**模块耦合度审计（Coupling Audit）**。通过构建清晰的“代码关系图谱（CodeGraph）”，帮助开发者快速理清项目架构，发现潜在的设计缺陷和废弃代码。

---

## 一、 代码图谱分析核心原则

进行代码图谱分析时，AI 助手必须遵循以下三项核心法则，以确保分析的客观性与准确性：

1. **绝对依据物理实体**：分析图谱中的所有节点（文件、类、函数）和连线（调用、依赖、包含）必须在代码库中有真实存在的物理文件或符号定义，**严禁脑补或臆造**任何不存在的代码关联。
2. **渐进式图谱构建**：面对庞大项目，应采用“自顶向下”与“自底向上”结合的策略。先划分高层模块边界（如文件夹拓扑），再深入核心类的调用链路，避免单张图谱承载过多信息导致混乱。
3. **高内聚低耦合审计**：图谱构建的最终目的是为了优化架构。在分析过程中，需敏锐捕获**循环依赖（Circular Dependencies）**、**高扇入扇出（High Fan-in/Fan-out）**及**上帝类/巨无霸函数（God Classes）**。

---

## 二、 图谱勘测与分析流程 (Exploration Flow)

AI 助手应当像雷达一样扫描整个项目，按照以下步骤建立代码图谱：

### 1. 结构大观：文件与目录层拓扑 (Directory & Package Level)
*   **动作**：使用 `list_dir` 或递归扫描工具，列出项目的所有主要模块、服务和组件文件夹。
*   **分析重点**：
    *   明确项目的架构模式（如经典的 MVC、三层架构、领域驱动设计 DDD 或是 Monorepo 单体多包结构）。
    *   梳理项目的入口文件（如 `server.py`, `app.js`, `index.tsx`, `main.go`）。

### 2. 依赖溯源：依赖导入流解析 (Dependency & Import Streams)
*   **动作**：使用 `grep_search` 扫描特定文件中的导入声明。
    *   *JavaScript/TypeScript*：正则匹配 `import ... from ...` 或 `require(...)`。
    *   *Python*：正则匹配 `import ...` 或 `from ... import ...`。
    *   *Go*：提取 `import (...)` 块。
*   **分析重点**：
    *   提取第三方核心库（如 React, Express, PyTorch, FastAPI）和本地模块的边界线。
    *   确定公共依赖层（Utilities/Common Helpers）的流向。

### 3. 调用追踪：符号与类调用图 (Symbol Call Graph)
*   **动作**：针对某个核心类（Class）或核心业务函数，全局搜索该符号被谁引入并调用（Caller），以及它本身调用了哪些下级服务（Callee）。
*   **分析重点**：
    *   绘制出一条从“用户入口/API控制器”到“底层数据库读写”的完整调用骨架链路。

---

## 三、 图谱可视化设计标准 (Visualization Standards)

将分析出的代码关系输出为 Mermaid 流程图时，AI 助手必须确保图表具有**极高的易读性**与**零语法错误**：

### 1. Mermaid 语法防错规则
*   **禁止包含特殊字符**：节点 ID 和文本标签中，严禁包含小括号 `()`、中括号 `[]`、花括号 `{}`、双引号 `"`、逗号 `,` 等 Mermaid 关键字。如果必须包含，**必须使用双引号将整个标签括起来**。
    *   *❌ 错误示范*：`A[getUser(id)] --> B[db.query]` (括号会破坏 Mermaid 语法)
    *   *✅ 正确示范*：`A["getUser(id)"] --> B["db.query()"]`
*   **防循环报错**：图谱连线如果出现闭环，确保箭头的方向不会引起渲染器的死循环。

### 2. 节点与连线视觉规范
为了使 Mermaid 图表在 Markdown 中美观优雅，使用以下标注规范：
*   **方括号 `[]`**：用于表示**物理文件或目录**（如 `[server.py]`）。
*   **圆角矩形 `()`**：用于表示**类或对象实例**（如 `(UserController)`）。
*   **体育场形 `([])`**：用于表示**核心功能函数/公共 API 接口**（如 `([loginUser])`）。
*   **连线样式**：
    *   实线 `-->`：表示强调用关系、直接的函数调用或类实例化。
    *   虚线 `-.->`：表示弱引用、事件订阅/发布、异步消息队列通信。
    *   双向箭头 `<-->`：用于突出警示**循环依赖缺陷**。

## 四、 代码图谱优化与质量红线 (Red Lines)

在对代码图谱进行审计时，以下指标被定义为**质量红线**。一旦在项目图谱中检测到，AI 助手应立即在其分析报告中以 `> [!WARNING]` 或 `> [!CAUTION]` 格式高亮预警并给出重构方案：

- [ ] **红线 1：严禁循环依赖 (Circular Dependencies)**
  *   *现象*：A 导入 B，B 导入 C，C 又导入 A。
  *   *后果*：极易引发编译时死锁、运行时加载失败（如 Node.js 的 `undefined` 导出）或垃圾回收内存泄漏。
- [ ] **红线 2：严禁跨层越权调用 (Layer Bypass)**
  *   *现象*：前端路由或 Web API 控制层绕过业务逻辑服务（Service Layer），直接对底层 ORM 或数据库驱动（Data Access Layer）进行读写。
  *   *后果*：彻底破坏分层架构，导致业务规则散落，极难维护。
- [ ] **红线 3：杜绝“上帝模块” (God Modules)**
  *   *现象*：某个文件或类的扇入（Fan-in）和扇出（Fan-out）数量极大，在图谱中呈现出极其密集的“海胆状”网络，所有核心链路全部交织在其身上。
  *   *后果*：该模块一旦发生微小改动，整个项目都面临瘫痪风险，必须进行职责解耦拆分。
